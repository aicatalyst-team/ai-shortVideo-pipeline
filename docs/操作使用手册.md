## 一、飞书机器人接入

> 系统通过飞书自建应用接收创作指令并把成片回传飞书会话。下面是从零到能收发消息的完整流程。

### 1.1 创建飞书自建应用

1. 登录 [飞书开放平台](https://open.feishu.cn) → **开发者后台** → **创建企业自建应用**
2. 填应用名称（如 `myAiVideos Bot`）、描述、图标
3. 创建完拿到两个关键值：
   - **App ID**（`cli_xxxxxxx`）
   - **App Secret**（`xxxxxxxx...`）

### 1.2 添加机器人能力

应用详情页 → **添加应用能力** → 选择 **机器人**。

### 1.3 配置事件订阅（Webhook）

应用详情页 → **事件订阅** → **请求地址**。

填入你的服务公网地址，路径为 `/webhook/feishu`：

```
https://your-domain.com/webhook/feishu
```

> **本地开发**：飞书后台**必须填公网地址**，本地不可用 `localhost`。可用 [cpolar](https://www.cpolar.com/) / [ngrok](https://ngrok.com/) / [frp](https://github.com/fatedier/frp) 做内网穿透：
> ```bash
> # cpolar 示例
> cpolar http 8000
> # 拿到 https://xxxx.cpolar.io，填进飞书后台
> ```

**URL 验证流程**：飞书会向上面这个地址 POST 一个 `{"type":"url_verification","challenge":"xxx"}`，服务端必须把 `challenge` 原样回传。我们的 `api/webhooks.py` 已实现这个逻辑：

```python
if body.get("type") == "url_verification":
    return {"challenge": body.get("challenge", "")}
```

点击 **保存**，飞书后台显示"验证成功"即配置完成。

### 1.4 订阅消息事件

**事件订阅** → **添加事件** → 勾选：

| 事件 | 说明 |
|---|---|
| `im.message.receive_v1` | 接收用户消息（必选） |
| `im.message.message_read_v1` | 用户已读回执（可选） |

### 1.5 配置权限

**权限管理** → 申请以下权限：

| 权限 | 用途 |
|---|---|
| `im:message` | 接收/发送消息 |
| `im:message.group_at_msg` | 群里 @机器人 消息 |
| `im:message.p2p_msg` | 单聊消息 |
| `im:resource` | 上传图片、视频、文件（成片回传需要） |
| `im:chat` | 读取群信息（可选） |
| `contact:user.id:readonly` | 解析 user_id（可选） |

### 1.6 发布应用

**版本管理与发布** → 创建版本 → 提交发布 → 管理员审批通过。

### 1.7 拉机器人进群 / 单聊

- **群**：群设置 → 添加机器人 → 选择刚发布的应用
- **单聊**：在飞书搜索应用名称 → 直接对话

### 1.8 配置 `.env`

```bash
FEISHU_APP_ID=cli_xxxxxxxxxxxxxxxx
FEISHU_APP_SECRET=xxxxxxxxxxxxxxxxxxxxxxxxxxxx
MY_FEISHU_USER_ID=ou_xxxxxxxxxxxxxxxx   # 可选,用于把消息只回给某个特定用户
```

`MY_FEISHU_USER_ID` 在群里 @机器人发"我的id"或在飞书开放平台 → 通讯录里能查到。

### 1.9 验证接入

重启 orchestrator 让新 .env 生效：

```bash
docker compose up -d --force-recreate orchestrator worker
```

在飞书会话里给机器人发 `你好`，看到机器人回复指令说明即接入成功。

容器日志能看到：

```
[Webhook] msg_type=text cid=oc_xxx content_keys=['text']
[Webhook] text=你好
[飞书] 发送 指令说明：...
```

### 1.10 常见指令

| 指令 | 行为 |
|---|---|
| `解说 <话题>` | 生成热搜解说类视频 |
| `科普 <话题>` | 生成知识科普类视频 |
| `故事 <话题>` | 生成情感故事类视频 |
| `奇闻 <话题>` | 生成奇闻趣事类视频 |
| `观点 <话题>` | 生成观点表达类视频 |
| `热搜` | 列出当前热搜话题 |
| `批量 <N>` | 批量生成 N 个视频 |
| `角色 <角色名>` | 切换角色 IP |
| `风格 <模板名>` | 切换风格模板 |
| `确认 <方案号>` | 确认 LLM 给出的方案进入下一步 |
| `继续` | 单段生成完毕后继续下一段 |
| `帮助` | 列出所有指令 |

## 二、测试

### 2.1 测试分类

| 类型 | 范围 | 是否要 docker | 是否花钱 |
|---|---|---|---|
| Python 单元测试 | 各模块独立逻辑、mock LLM | ❌ | ❌ |
| Python 集成测试 | 数据库 + Redis 真实交互 | ✅ | ❌ |
| Java 单元测试 | 网关 filter / service / controller | ❌ | ❌ |
| Java 压测（k6） | 限流 / 熔断 / failover | ✅ | ❌ |
| 端到端（E2E） | 完整管线含真实 AI 调用 | ✅ | ✅（少量） |

### 2.2 Python 测试

```bash
# 全量回归
docker compose exec -T orchestrator pytest tests/ -q

# 跑特定测试文件
docker compose exec -T orchestrator pytest tests/test_av_sync_rescue.py -v

# 跑特定 case
docker compose exec -T orchestrator pytest tests/test_clip_consistency.py::test_check_consistency_returns_full_result_when_passed -v

# 显示打印输出
docker compose exec -T orchestrator pytest tests/ -v -s

# 跑覆盖率
docker compose exec -T orchestrator pytest tests/ --cov=layers --cov=core --cov=api
```

写新测试的约定：

- 单元测试 mock 掉所有外部 API（LLM、TTS、Kling）
- 用 `monkeypatch` 注入假返回，参考 `tests/test_clip_consistency.py`
- 异步用 `@pytest.mark.asyncio`

### 2.3 Java 测试

```bash
cd gateway

# 单元 + 集成
mvn test

# 跑特定模块
mvn test -Dtest=JwtServiceTest

# 跳过测试 build
mvn package -DskipTests

# 看测试报告
ls target/surefire-reports/
```

### 2.4 压测（k6）

需要 [k6](https://k6.io/docs/get-started/installation/) 安装在宿主机：

```bash
# 限流压测
k6 run gateway/tests/ratelimit_k6.js

# 自定义 VUs（并发用户）和持续时间
k6 run --vus 50 --duration 30s gateway/tests/ratelimit_k6.js
```

期望看到：

- 正常并发下大部分请求 200
- 超过限流阈值时出现 429
- 平均延迟稳定（无指数级增长）

### 2.5 手动 API 测试

#### 健康检查

```bash
curl -s http://localhost:8000/health | jq
curl -s http://localhost:8080/actuator/health | jq
```

#### 模拟飞书消息（不接真飞书也能测主流程）

```bash
curl -X POST http://localhost:8000/webhook/feishu \
  -H "Content-Type: application/json" \
  -d '{
    "type": "event_callback",
    "event": {
      "message": {
        "message_type": "text",
        "chat_id": "test_chat_001",
        "message_id": "test_msg_001",
        "content": "{\"text\":\"科普 LLM 是什么\"}"
      }
    }
  }'
```

#### 跨语言 trace 透传验证

```bash
# 带自定义 trace_id 调网关 → Python 应该看到同一 trace
curl -si -H "X-Trace-Id: my-test-trace-001" \
  -H "Authorization: Bearer $TOKEN" \
  http://localhost:8080/api/v1/storyboards/PLAN_ID

# 在两边日志找
grep "trace=my-test-trace-001" data/logs/*.log
docker compose logs gateway orchestrator | grep "my-test-trace-001"
```

#### 限流触发

```bash
# 用 ab 或 wrk 短时间打几十次同一端点
ab -n 100 -c 20 -H "Authorization: Bearer $TOKEN" \
   http://localhost:8080/api/v1/clips/X/regenerate

# 期望部分请求收到 HTTP 429 + Retry-After
```

### 2.6 前端 Mock 测试

```bash
cd sl-vue

# Linux/Mac
VITE_USE_MOCK=true npm run dev

# Windows PowerShell
$env:VITE_USE_MOCK = "true"; npm run dev
```

打开 <http://localhost:5173/canvas/MOCK001> 看 mock 数据展示，**不打后端**。

### 2.7 端到端冒烟（真发一条）

> ⚠️ 真实调用 AI 服务，会产生少量费用（约 ¥0.3-2 / 条）。

1. 确认 `.env` 内所有 API key 已配
2. 启动全栈：`docker compose up -d`
3. 飞书发 `科普 LLM 是什么`
4. 跟踪进度：`tail -f data/logs/orchestrator.log`
5. 等待飞书回传成片视频（约 5-10 分钟）

预期日志关键节点：

```
[Webhook] text=科普 LLM 是什么
[垂类链] theme=... style=knowledge_explainer
[审核链] GLM审核
[评估链] 开始评估
[飞书] 发送 解说视频制作方案
[文生图] 提交 model=kling-v1-5
[Kling] 提交 model=kling-v2-5-turbo
[TTS] 完成
[av_sync] report: ✅ 视频 ... / 配音 ...
[飞书视频] 上传 ...
```

## 三、日常维护

### 3.1 健康检查清单（每日）

```bash
# 一行串联三个端点
curl -fs http://localhost:8000/health \
  && curl -fs http://localhost:8080/actuator/health \
  && docker compose ps --format json | jq -r '.[] | "\(.Name): \(.State)"'
```

或写成 crontab：

```cron
# 每 5 分钟自动探活,失败发邮件
*/5 * * * * curl -fs http://localhost:8000/health > /dev/null || echo "orchestrator down" | mail -s "ALERT" you@example.com
```

### 3.2 日志监控

```bash
# 最近 10 分钟的错误
docker compose logs --since 10m orchestrator gateway worker \
  | grep -E "ERROR|Traceback|Exception" | tail -30

# 看日志归档大小（防爆盘）
du -sh data/logs/

# 找具体一条请求的完整链路
grep "trace=b09df48c0b714426" data/logs/*.log
```

**告警关键字**（建议接 ELK / Grafana Loki）：

- `ERROR`、`Traceback`、`CircuitBreaker.*OPEN`
- `AVDriftTooLargeError`、`av_rescue.*failed`
- `Langfuse.*disabled`、`anchors.*extract failed`

### 3.3 数据库维护

#### 备份

```bash
# 单次备份
docker compose exec -T postgres pg_dump -U postgres myaivideos | gzip > backup-$(date +%F).sql.gz

# 加进 crontab(每天 3 点)
0 3 * * * docker compose exec -T postgres pg_dump -U postgres myaivideos | gzip > /backup/myaivideos-$(date +\%F).sql.gz && find /backup -name "myaivideos-*.sql.gz" -mtime +14 -delete
```

#### 恢复

```bash
gunzip -c backup-2026-05-30.sql.gz | docker compose exec -T postgres psql -U postgres myaivideos
```

#### 表统计 + 优化

```bash
# 进 psql
docker compose exec -T postgres psql -U postgres -d myaivideos

# 看大表
SELECT relname, pg_size_pretty(pg_total_relation_size(relid))
  FROM pg_catalog.pg_statio_user_tables
 ORDER BY pg_total_relation_size(relid) DESC LIMIT 10;

# vacuum + analyze（手动触发,Postgres 平时自动跑）
VACUUM ANALYZE;

# 看慢查询(需要 pg_stat_statements 扩展)
SELECT query, calls, total_exec_time, mean_exec_time
  FROM pg_stat_statements
 ORDER BY mean_exec_time DESC LIMIT 10;
```

### 3.4 Redis 监控

```bash
# 队列堆积情况
docker compose exec -T redis redis-cli LLEN arq:queue

# 内存使用
docker compose exec -T redis redis-cli INFO memory | grep used_memory_human

# 慢查询日志
docker compose exec -T redis redis-cli SLOWLOG GET 10
```

### 3.5 容器 / 镜像清理

```bash
# 看磁盘
df -h
docker system df

# 清未使用的镜像/容器/网络
docker system prune -af

# 单独清 build cache
docker builder prune -af

# 清旧的成片(超过 30 天)
find output -name "*.mp4" -mtime +30 -delete
```

### 3.6 模型成本监控

```bash
# 当日各租户/模型成本
docker compose exec -T postgres psql -U postgres -d myaivideos -c "
SELECT tenant_id, model, total_calls, total_cost_cny, fallback_calls
  FROM billing_daily
 WHERE dt = CURRENT_DATE
 ORDER BY total_cost_cny DESC;
"

# 通过网关 API 查
curl -s http://localhost:8080/api/v1/usage/today \
  -H "Authorization: Bearer $TOKEN" | jq

# 成本异常告警(建议接 Prometheus rule)
# 例: 单日单租户成本 > ¥50 触发告警
```

### 3.7 Langfuse 链路检查（如启用）

- 登录 [cloud.langfuse.com](https://cloud.langfuse.com) → Traces
- 按 trace_id / session_id / tag 过滤
- 看每条调用的输入/输出/耗时/成本
- 找异常 trace（耗时 > P99、报错、failover 等）

## 四、安全

### 4.1 JWT 密钥轮换（每 90 天）

```bash
# 1. 生成新密钥
NEW=$(openssl rand -base64 32)

# 2. 写入 .env
sed -i "s|^GATEWAY_AUTH_JWT_SECRET=.*|GATEWAY_AUTH_JWT_SECRET=${NEW}|" .env

# 3. 重启网关让新密钥生效
docker compose up -d --force-recreate gateway

# 4. 通知所有客户端用新密钥重签 token,旧 token 即时失效
```

### 4.2 AI 服务 API key 轮换

各供应商后台轮换后，同步更新 `.env`，然后：

```bash
docker compose up -d --force-recreate orchestrator worker gateway
```

### 4.3 不进 git 的文件清单

- `.env`（所有 secret）
- `data/logs/`（含敏感 trace）
- `output/`（生成内容）
- `data/hf_cache/`（模型缓存）
- 任何含 token / API key 的文档

### 4.4 端口暴露原则

| 端口 | 生产建议 |
|---|---|
| 80 / 443 | 公网（仅前端 + 必要 API） |
| 8080（网关） | **不直接对公网**，走反向代理 |
| 8000（FastAPI） | **不对公网**，仅内网 |
| 5432 / 6379 / 9000 / 9001 | **不对公网**，绑 127.0.0.1 或内网 IP |

### 4.5 定期安全扫描

```bash
# 用 gitleaks 扫历史里的泄密
gitleaks detect --source . --verbose

# 用 trivy 扫镜像漏洞
trivy image ai-platform-app:latest
trivy image ai-platform-gateway:latest
```

## 五、升级流程

### 5.1 代码升级（小版本）

```bash
# 1. 拉新代码
git pull

# 2. 看 commit message 判断需不需要 build / migration
git log --oneline -10

# 3. Python 改了 → 重 build
docker compose build orchestrator

# 4. Java 改了 → 重 build
docker compose build gateway

# 5. 用新镜像重建容器
docker compose up -d --force-recreate --no-build orchestrator worker gateway

# 6. 跑迁移(如果新增了表/字段)
docker compose exec -T orchestrator alembic upgrade head

# 7. 对账
git rev-parse HEAD                              # 应等于远程
docker images ai-platform-app --format '{{.CreatedAt}}'    # 应是刚才
curl -s http://localhost:8000/health
curl -s http://localhost:8080/actuator/health
```

### 5.2 回滚

```bash
# Python
cd /path/to/repo
git log --oneline -10
git checkout <旧 SHA>
docker compose build orchestrator
docker compose up -d --force-recreate --no-build orchestrator worker

# 数据库回滚谨慎(多数 migration 不可逆):
docker compose exec -T orchestrator alembic stamp <旧版本号>
```

### 5.3 数据库 schema 变更

```bash
# 1. 写新 migration
docker compose exec -T orchestrator alembic revision -m "add foo column"

# 2. 编辑生成的 versions/xxx_add_foo_column.py

# 3. 本地验证
docker compose exec -T orchestrator alembic upgrade head
docker compose exec -T orchestrator alembic downgrade -1   # 验证可回滚
docker compose exec -T orchestrator alembic upgrade head   # 再升回去

# 4. 跑测试确认
docker compose exec -T orchestrator pytest tests/ -q

# 5. 提交 + 部署
git add db/migrations/versions/
git commit -m "feat: add foo column"
```
